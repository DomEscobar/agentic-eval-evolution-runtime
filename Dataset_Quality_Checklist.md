# Dataset Quality Checklist

Use this checklist before calling any EvalSpec-generated dataset "good".

The rule:

```text
A good schema does not make a good dataset.
Cases, oracles, fixtures, splits, hidden tests, and review quality make a good dataset.
```

This checklist applies to:

- QA datasets
- golden datasets
- regression suites
- redteam sets
- API test suites
- RAG eval sets
- coding-agent patch-loop benchmarks

## Quality Verdict Levels

Use these labels in reports.

### Level 0: Schema Only

The project has an EvalSpec or endpoint list, but no generated cases.

Do not use for claims.

### Level 1: Generated Draft

Cases are generated, but not reviewed.

Use for exploration only.

### Level 2: Reviewed Validation Set

Cases have deterministic fixtures and oracles, and at least one reviewer or verifier agent has checked them.

Use for candidate selection.

### Level 3: Protected Holdout

Cases are hidden from mutators and coding agents. Detailed failure feedback is not exposed.

Use for release gates.

### Level 4: Production-Representative Benchmark

Cases reflect real usage distribution, edge cases, regressions, and failures from production or domain review.

Use for credible product-quality claims.

### Level 5: Contamination-Resistant Benchmark

Holdout/redteam are private, versioned, rotated, deduplicated, and checked for leakage.

Use for external claims or agent-to-agent comparisons.

## Dataset Audit

### 1. Task Validity

Each case must map to a real task, product requirement, or domain rule.

Pass criteria:

- task is meaningful
- task is solvable with available context
- task is not ambiguous unless ambiguity is explicitly being tested
- task belongs to an expected category
- task has a clear failure mode

Reject cases that:

- test implementation trivia
- test impossible behavior
- depend on hidden assumptions not in the spec
- have no user or system value

### 2. Outcome Validity

Each case must define what correct means.

Pass criteria:

- expected output or expected state is explicit
- oracle is deterministic, or judge rubric is clear
- bad outputs fail
- alternative correct outputs are allowed when appropriate
- expected side effects are defined

Reject cases that:

- only assert HTTP 200
- only assert "looks good"
- depend on snapshot text without semantic checks
- have expected outputs generated by the same unverified agent with no review

### 3. Oracle Quality

Every case needs at least one oracle.

Good oracles:

- invariant checks
- exact state checks
- schema checks
- permission checks
- source/citation checks
- hidden unit tests
- calibrated judge rubric

Weak oracles:

- string similarity only
- generic LLM judge with no rubric
- status-code-only checks
- "response contains something"

Hard rule:

```text
No oracle, no dataset case.
```

### 4. Fixture Determinism

The same case should produce the same expected result after reset.

Pass criteria:

- reset command exists
- seed command exists
- known entities are stable
- time is frozen or controlled
- randomness is seeded
- external services are mocked or explicitly marked live

Reject cases that:

- depend on production data
- depend on current time without freezing time
- require live third-party systems without a live-dependency label
- mutate shared state without cleanup

### 5. Coverage

A useful dataset covers more than happy paths.

Required categories:

- happy path
- edge case
- negative case
- permission/security case
- regression case
- cross-step workflow

For RAG:

- answerable questions
- unanswerable questions
- stale/outdated source traps
- citation requirements
- conflicting sources
- missing context

For APIs:

- valid inputs
- invalid inputs
- permission failures
- state transitions
- idempotency
- mutation vs read-only behavior
- rate/limit boundaries

For coding agents:

- simple bug fix
- multi-file bug fix
- regression-sensitive patch
- forbidden-file trap
- hidden-test-only failure
- minimality/diff-size pressure

### 6. Split Hygiene

The split policy must be explicit.

```text
train       visible to mutator
validation  candidate selection
holdout     final gate only
redteam     adversarial/regression gate
```

Pass criteria:

- each case has exactly one split
- holdout is hidden from optimizer prompts
- redteam is hidden or access-controlled
- no duplicate cases across splits
- no near-duplicate leakage across splits
- failure details for holdout are summarized, not shown in full to the mutator

Reject datasets where:

- the optimizer sees holdout cases
- holdout failures are fed back as detailed examples
- train and holdout differ only by wording
- generated variants of the same task are spread across splits

### 7. Contamination Resistance

A dataset exposed repeatedly becomes less trustworthy.

Pass criteria:

- public benchmark cases are not used for final product claims
- project-local holdout remains private
- holdout version is pinned
- split hashes are included in reports
- redteam cases are rotated periodically
- production failures feed train/validation first, not holdout

For external claims, require:

- private holdout
- independent review
- deduplication against public examples
- clear model/tool versions
- reproducible evaluation report

### 8. Reviewer Confidence

Generated datasets need review.

Minimum:

- verifier agent review for every case
- human spot review for each category
- full human review for holdout/redteam in high-stakes domains

Track:

- reviewer id
- review timestamp
- confidence
- reason for changes
- unresolved ambiguity

### 9. Guardrail Coverage

Guardrail cases should be disqualifying, not weighted.

Required checks:

- hidden tests cannot be changed
- evaluator cannot be changed
- archive cannot be changed
- secrets cannot be read
- forbidden paths cannot be modified
- live production systems cannot be called unless explicitly allowed
- unsafe output patterns are rejected

### 10. Reporting Quality

Every dataset report should include:

- dataset version
- EvalSpec version
- split counts
- category coverage
- oracle coverage
- fixture commands
- reviewer summary
- known gaps
- contamination/leakage checks
- holdout access policy
- redteam rotation policy

## Dataset Quality Score

Use this score to audit the dataset, not the system under test.

```text
25% oracle clarity
20% production representativeness
15% split hygiene
15% edge/negative/security coverage
10% fixture determinism
10% reviewer confidence
 5% contamination resistance
```

Hard failures:

- any case without an oracle
- holdout visible to optimizer
- hidden tests editable by coding agent
- no reset/seed for stateful system
- production secrets required for evaluation

## Agent Instructions

When an agent is asked to create a dataset from EvalSpec:

1. Read `EvalSpec_Guide.md`.
2. Generate candidate cases.
3. Attach at least one oracle to every case.
4. Assign a category and split.
5. Check for duplicates and near-duplicates.
6. Reject cases without deterministic setup.
7. Produce a dataset quality report.
8. Mark the dataset level from 0 to 5.
9. Do not call the dataset "good" below Level 3.
10. Do not use a dataset for external claims below Level 5.

Expected output files:

```text
datasets/train.jsonl
datasets/validation.jsonl
datasets/holdout.jsonl
datasets/redteam.jsonl
reports/dataset_quality_report.md
reports/split_leakage_report.json
reports/oracle_gap_report.json
reports/coverage_report.json
```
